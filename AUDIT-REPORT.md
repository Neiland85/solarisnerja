## 2. HECHOS OBSERVABLES

### Arquitectura
- **Runtime**: No se declara `export const runtime = 'edge'` en ninguna API route. Todas ejecutan en Node.js runtime por defecto.
- **Pool PostgreSQL**: Singleton `pg.Pool` sin configuración de `max`, `idleTimeoutMillis`, `connectionTimeoutMillis`. Defaults de `pg`: max=10 conexiones.
- **Rate Limit**: `Map<string, Entry>` en memoria del proceso. No persiste entre instancias, workers o cold starts.
- **Idempotencia**: `UNIQUE INDEX (email, event_id)` en DB pero el `INSERT` no usa `ON CONFLICT DO NOTHING` — lanzará error en duplicados.
- **Middleware**: Solo inyecta `x-request-id`. No hay CORS, no hay CSP, no hay validación de origin.
- **Tests**: 3 archivos test (domain, rate-limit, contract). No hay tests de integración ni de la API route.
- **CI**: Typecheck + Lint + Build. **No ejecuta tests**.
- **Logger**: JSON a stdout/stderr. Sin correlation ID propagado, sin niveles configurables, sin destino externo.
- **Security headers**: X-Frame-Options, X-Content-Type-Options, Referrer-Policy. **Falta CSP, HSTS, Permissions-Policy**.
- **Layout metadata**: Todavía dice "Create Next App" / "Generated by create next app".
- **Tracking**: `console.log` de eventos en producción. `(window as any)` — sin tipado.

### GDPR
- Se captura `email` (dato personal) vinculado a `event_id`.
- No hay registro de consentimiento explícito.
- No hay campo `consent_timestamp`, `consent_source`, `privacy_policy_version`.
- No hay política de retención definida.
- No hay endpoint de derecho de acceso/supresión (ARCO/RGPD Arts. 15-17).
- No hay registro de actividades de tratamiento (Art. 30 RGPD).
- No hay aviso de privacidad vinculado al formulario.

## 3. ANÁLISIS TÉCNICO

### 3.1 Arquitectura Edge-First

**Veredicto**: El proyecto **no es Edge-First**. Es Node.js-first con cache headers configurados para CDN.

**Evidencia**:
- Ninguna route declara `runtime = 'edge'`.
- Usa `pg` (TCP sockets) que es incompatible con Edge runtime de Vercel.
- El rate limiter usa `Map` en memoria del proceso Node.js.

**Esto no es necesariamente malo** para el caso de uso actual. Pero la narrativa "Edge-First" no se corresponde con la implementación.

| Aspecto | Estado | Riesgo |
|---|---|---|
| Runtime declarado | Node.js (default) | Bajo — correcto para pg |
| Pool de conexiones | Singleton sin config | **Alto** — en serverless, sin `max`/`idle` causa connection exhaustion |
| Rate limit in-memory | `Map` por proceso | **Crítico** — inútil con >1 instancia/worker |
| Idempotencia DB | Index unique, pero INSERT sin ON CONFLICT | **Alto** — duplicados causan error 500 |
| Cold start risk | Pool se crea lazy en primera request | Medio — primera request lenta |
| Concurrencia | Sin transacciones, sin row-level locking | Bajo para volumen actual |

### 3.2 Calidad de Código

**Estructura de carpetas**: Buena separación intencional (`domain/`, `adapters/`, `contracts/`, `lib/`, `ui/`). Sigue principios DDD-lite de forma correcta.

**Tipado**: TypeScript strict habilitado con `strictNullChecks`, `noUncheckedIndexedAccess`, `noImplicitOverride`. **Excelente configuración**.

| Aspecto | Evaluación |
|---|---|
| Cohesión | Alta — cada módulo tiene responsabilidad clara |
| Acoplamiento | Medio — `route.ts` importa directamente `saveLead`, no inyecta dependencia |
| SRP | OK en domain; violación leve en `route.ts` (valida + rate-limit + honeypot + save) |
| DIP | **Violado** — `lead-repository.ts` importa `pool.ts` directamente, sin interfaz |
| Naming | Consistente. `camelCase` TS, `snake_case` SQL. Alineado con convenciones |
| Error handling | `catch` genérico en route sin logging del error real |

**Anti-patterns detectados**:

1. **Catch-and-swallow**: Línea 66-76 de `route.ts` — `catch (error)` no registra `error.message` ni `error.stack`. Imposible diagnosticar fallos en producción.

2. **LeadInput duplicado**: El tipo `LeadInput` se define localmente en `route.ts` Y se infiere del schema JSON. No hay source of truth único.

3. **No inversión de dependencias**: `createLead` → `saveLead` se orquesta en la route sin inyección. Si se necesitan tests de integración, hay que mockear módulos enteros.

4. **Rate limiter leaks memory**: El `Map` crece indefinidamente. No hay cleanup de entries expiradas. En producción prolongada = memory leak.

5. **`(window as any)`** en `tracking.ts`: Casting inseguro que anula TypeScript.

## 5. DEUDA TÉCNICA ESTRUCTURAL

### Causas Raíz (no síntomas)

1. **Ausencia de inversión de dependencias**: El dominio (`createLead`) no define puertos. La orquestación en `route.ts` conecta directamente implementaciones concretas. Consecuencia: tests de integración imposibles sin mockear módulos, y cambiar de PostgreSQL requiere reescribir el adapter Y la route.

2. **Validación no genera tipos**: El JSON Schema valida en runtime pero el tipo `LeadInput` se define a mano en `route.ts`. Si el schema cambia y el tipo no, divergen silenciosamente. Falta `json-schema-to-ts` o codegen.

3. **Sin capa de aplicación/use-case**: La route HTTP orquesta directamente dominio + infra. Esto impide reutilizar la lógica de "crear lead" desde otro trigger (cron, queue, otro endpoint).

4. **Observabilidad ad-hoc**: Logger básico sin trace propagation, sin métricas, sin health checks con degradación parcial (readyz solo chequea DB, no rate-limiter ni servicios downstream).

## 11. RIESGOS RESIDUALES

Incluso completando todas las fases:

1. **Dependencia de Vercel/hosting TLS**: Si el despliegue no fuerza HTTPS, los datos viajan en claro. Mitigación: HSTS + redirect HTTP→HTTPS.
2. **Single point of failure en PostgreSQL**: Sin réplica ni failover. Mitigación: managed DB con HA (Neon, Supabase, RDS Multi-AZ).
3. **Third-party widget (Ticketmaster)**: Script externo puede cambiar sin aviso, inyectar tracking, o fallar. Mitigación: SRI hash (si disponible), fallback graceful, CSP para limitar dominios.
4. **Email como identificador**: Sin verificación de propiedad del email. Cualquiera puede registrar emails ajenos. Mitigación: double opt-in (largo plazo).
