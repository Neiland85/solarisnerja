## 3. Deuda Técnica Estructural

### Causa Raíz 1: Auth como afterthought

El sistema de autenticación se diseñó como parche, no como capa transversal.

- **Síntoma**: 9/18 rutas admin desprotegidas, eventos CRUD abierto
- **Causa**: No hay middleware de auth centralizado. Cada ruta debe importar `requireAdmin` manualmente — y muchas no lo hacen.
- **Evidencia**: `proxy.ts` solo protege `/dashboard` (redirect a login), pero no `/api/admin/*`
- **Impacto**: Cualquier endpoint olvidado queda expuesto permanentemente
- **Artefacto muerto**: `session.ts` exporta `createSessionToken()` que ya no se usa en ningún sitio

### Causa Raíz 2: Infraestructura in-memory en entorno serverless

Todo el state (queue, cache, métricas, audit, circuit breaker) vive en variables de módulo. Vercel serverless hace cold starts frecuentes → **pérdida total de estado**.

- **Síntoma**: Queue pierde leads, métricas se resetean, audit trail desaparece
- **Causa**: Se diseñó para proceso long-lived pero se deploya en funciones serverless
- **Evidencia**: `burstQueue.ts` (array), `swr.ts` (Map), `metricsCollector.ts` (Map), `auditLog.ts` (array de 1000)
- **Redis existe** (`ioredis` en dependencies, `redisQueue.ts` implementado) pero nunca se conectó al flujo principal

### Causa Raíz 3: Pool DB sin resiliencia

- **Síntoma**: Errores no capturados en repositorios, pool size insuficiente
- **Causa**: `getPool()` crea pool una vez sin retry ni health-check. Repos hacen `pool.query()` sin try/catch.
- **Evidencia**: `lead-repository.ts`, `event-repository.ts` — ninguna función tiene error handling propio
- **Combo**: `rejectUnauthorized: false` + pool max=5 + zero retry = frágil
